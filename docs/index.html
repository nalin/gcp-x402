x402 · BigQuery · USDC

Public data, no account required.

gcp-x402 lets an AI agent query Google BigQuery's public datasets without a Google Cloud account — paying a few thousandths of a dollar per query, automatically, in stablecoin.

BigQuery hosts ~200 free public datasets — census records, every Hacker News post, the Ethereum blockchain, decades of weather. The data is free, but running a query isn't: Google bills the compute (about $6.25 per terabyte scanned) to whoever's project runs the job. To get such a project you need a billing account and a credit card. An autonomous agent has neither.

The whole thing in one sentence
gcp-x402 is a toll booth in front of BigQuery: the agent sends a question, gets back an exact price, pays it in USDC over a web standard called x402, and gets its rows — while a hosted proxy quietly owns the Google Cloud bill and pockets a small margin.
A plain analogy
Imagine a library with priceless books that are free to read, but the reading room charges by the page you actually turn. You don't need a membership card — you drop a coin in the slot for exactly the pages you read, and a clerk (the proxy) settles up with the library on your behalf. gcp-x402 is that coin slot, for data.

The core idea

Almost every paywall charges a fixed price. gcp-x402 can't — the cost of a BigQuery query is unknowable until you look at it. So the price is computed per query, and that single fact shapes the entire design.

BigQuery offers a dry run: ask it to analyze a query without running it, and it returns the exact number of bytes the real query would be billed for — for free, in well under a second. That dry run is the keystone. It does two jobs at once:

Job 1 — pricing oracle

How much will this cost?

Bytes scanned → dollars → a USDC price, quoted before the agent pays a cent. See pricing →

Job 2 — safety oracle

Is this query allowed?

The dry run also lists every table the query touches, so the proxy can reject anything that isn't read-only public data. See security →

Architecture

Three actors. The agent holds a wallet and asks questions; the proxy prices, guards, runs, and charges; BigQuery does the compute. Payment settles on the Base blockchain through a neutral “facilitator.”

How a query works

A single endpoint, POST /api/query, drives a two-step handshake defined by the x402 standard. The first call comes back with a price; the second call carries the payment.

  1. Agent sends the query — no payment yet.
    POST /api/query { sql }
  2. Proxy dry-runs it to learn the exact bytes and tables.
    BigQuery → 110,355,534 bytes · referencedTables[]
  3. Proxy checks safety, prices it, and replies “402 Payment Required”.
    402 + signed quote { bytes, price, expiry }
  4. The agent's wallet signs a USDC payment for that price and retries.
    POST /api/query { sql } + X-PAYMENT (signed authorization)
  5. Proxy re-checks the price, verifies the payment, runs the real query.
    maximum_bytes_billed = quote → a runaway query fails free
  6. Only after the query succeeds, the payment is settled on-chain. Rows are returned.
    200 OK + rows + settlement receipt
Why two calls?
That's just how x402 works: the server quotes a price with a 402 status, and the client pays and retries. It means the agent always sees the price before spending anything, and a query that fails is never charged.
▸ Deep dive: the exact server logic code

The interesting part is what no off-the-shelf payment middleware does: pricing each request from a dry run, and binding the payment to the specific query.

// 1. QUOTE — price the query from a free dry run
const dry   = await bq.dryRun(sql);
const bytes = dry.totalBytesProcessed;
const refs  = dry.referencedTables;

// 2. SAFETY — public, read-only tables only
for (const t of refs)
  if (t.projectId !== "bigquery-public-data") reject();

// 3. PRICE — cost × margin, with a floor (USDC)
const price = Math.max(billableBytes * USD_PER_BYTE * MARKUP, FLOOR);

// 4. No payment yet? Answer 402 with the price + a signed quote.
// 5. On the paid retry: re-price from the ACTUAL body, verify the
//    payment covers it, run with a hard byte cap, THEN settle.
await bq.run(sql, { maximumBytesBilled: cap });   // fails free if over

Ordering is deliberate — verify → execute → settle. Verification is cheap and happens before any BigQuery compute is spent; settlement (the actual movement of funds) happens last, only if the query succeeded.

The pricing oracle

BigQuery's dry run returns the precise byte count the real query will be billed for. The proxy turns that number into a price.

InputValueMeaning
Bytes scannedfrom dry runExact billable bytes — deterministic for static public tables.
Rate~$6.25 / TiBBigQuery's US on-demand price — what Google charges the proxy.
Markup2×The proxy's margin.
Floor$0.002Minimum, so tiny queries still clear payment overhead.
Per-table min10 MBMatches BigQuery's own minimum billing per referenced table.

So a query that scans 1 GiB costs the agent about $0.012; a 100 MB query hits the floor at $0.002. The price is quoted up front, every time.

▸ Deep dive: the “LIMIT doesn't help” gotcha cost model

BigQuery is columnar: it bills for the columns a query reads across all rows, not the rows it returns. Two consequences the agent is taught to respect:

  • Never SELECT * on a wide table — it scans every column and can cost 100× more than selecting one.
  • LIMIT does not reduce cost — it caps rows returned, not bytes scanned. To scan less, filter on a partition column or select fewer columns.

There's also a hard ceiling, MAX_BYTES_PER_QUERY (set to 1 GiB on the live deployment), so no single query — by the agent or anyone — can scan a giant table. A 3.4 TB SELECT * is rejected at the dry-run stage, before any scan.

Paying with x402

x402 revives the long-dormant HTTP status code 402 Payment Required as a real payment protocol for machines. No accounts, no API keys — just a wallet and an HTTP retry.

When the proxy answers 402, it includes payment requirements: the amount, the token (USDC), the network (Base), and where to send it. The agent's client signs an EIP-3009 “transfer with authorization” — a gasless way to authorize a USDC transfer — and retries the request with that signature in an X-PAYMENT header. A neutral facilitator service verifies the signature and broadcasts the settlement on-chain.

Gasless for the agent
The agent's wallet only needs USDC — no ETH for gas. The facilitator submits the on-chain transaction, so funding a new wallet is a single step.
▸ Deep dive: exact today, upto tomorrow protocol

The proxy ships the exact scheme: the agent pays exactly the quoted price. It's supported by every facilitator and is correct for immutable public datasets, where the dry-run byte count equals the billed byte count.

x402 v2 added an upto scheme — authorize a maximum, then settle the actual cost after the work runs. That's a perfect future fit for metered compute (a BigQuery cache hit bills ~0 bytes), and the code is structured so swapping it in is a localized change.

Why it can't be gamed

The attack to worry about: get a cheap quote on a tiny query, then swap in a petabyte SELECT * on the paid retry. Three independent layers stop it.

Layer 1 — authoritative

Re-price on the paid call

The proxy never trusts the first quote. It re-runs the dry run on the actual body it's about to execute. If the query grew, the new price no longer matches the authorized amount, and verification fails.

Layer 2 — tamper-evident

Signed quote

A short-lived quote, signed by the server, binds a price to a hash of the exact SQL. The agent's receipt — and a cheap tripwire if the body changed.

Layer 3 — backstop

Hard byte cap

The real query runs with maximum_bytes_billed set to the quote. A query that would exceed it fails for free instead of billing the proxy.

Plus — sandboxing

Read-only, public-only

DML/DDL/scripting are rejected; only bigquery-public-data tables pass the allowlist. The service account can do nothing but run query jobs.

▸ Deep dive: a small headroom on the byte cap gotcha

In live testing, a query priced at 110,355,534 bytes actually billed 111,149,056 — BigQuery rounds billed bytes up slightly past the dry-run estimate. An exact cap spuriously killed such jobs. The fix: a small buffer (default 20%) on the cap. The agent is still charged the quoted price; the proxy absorbs the sub-percent overage, which the 2× markup more than covers. The 1 GiB hard ceiling remains the real runaway guard.

Economics

The proxy is a reseller: it pays Google for bytes scanned and charges the agent that cost plus a margin.

Query sizeProxy's BigQuery costAgent pays (2× + floor)
100 MB~$0.0006$0.002 (floor)
1 GB~$0.006~$0.012
100 GB~$0.61~$1.22

BigQuery's first 1 TiB/month is free per billing account — a busy proxy burns that in days, so pricing is set at the marginal rate and the free terabyte is treated as upside. The floor exists because on-chain settlement has a small per-transaction overhead; a sub-cent floor keeps tiny queries from running at a loss.

The three pieces

One repository, three artifacts. The proxy is the backend; the client ships in two forms so any agent can use it.

1 · The proxy (server — Next.js on Cloud Run)

Holds the Google Cloud credentials and the receiving wallet. Exposes POST /api/query (the paid endpoint) and a free GET /api/datasets. This is the only component that touches BigQuery or the blockchain.

2 · The client (a dual-mode program)

The same small program runs two ways from one install:

no arguments

MCP server

Exposes structured tools (bigquery_query, bigquery_estimate, wallet_info…) to MCP-native clients.

with arguments

Plain CLI

The skill drives it over the shell: npx -y github:nalin/gcp-x402 query "<sql>" — no MCP setup needed.

3 · The skill (context for the agent)

A single Markdown file that teaches an agent how to use the tool well — the funding flow, the cost rules (no SELECT *, LIMIT doesn't help), read-only constraints, and worked examples. It triggers automatically when someone asks a question a public dataset could answer.

The agent wallet

On first use the client generates its own wallet — no keys to paste — and stores it per project.

  • Saved to ./.gcp-x402/wallet.json in the project directory, with 0600 permissions and an auto-written .gitignore so the private key can never be committed.
  • Each project gets its own wallet, so a fresh project means a fresh fund-me flow — and spending is isolated between projects.
  • A new wallet starts empty; the agent shows its address and asks the user to send a little USDC on Base. After that, queries pay themselves.
  • A configurable ceiling (MAX_PAYMENT_USD, default $1) caps what any single query may auto-pay.

Installing it

The package is public, so installs need no auth, no clone, no build step.

As a skill (simplest, per project)

# run from the project you want to enable
mkdir -p .claude/skills/bigquery-public-data && \
curl -fsSL https://gcp-x402-975410367881.us-central1.run.app/skill \
  -o .claude/skills/bigquery-public-data/SKILL.md

As an MCP server

claude mcp add gcp-x402 \
  --env PROXY_URL=https://gcp-x402-975410367881.us-central1.run.app \
  -- npx -y github:nalin/gcp-x402

Both point at the same hosted proxy and the same per-project wallet.

Status & roadmap

  • Live on testnet. End-to-end verified on Base Sepolia: a real query returned rows, paid $0.002, and settled on-chain — with no charge when a query fails.
  • Next: the upto scheme (charge actual billed bytes, so cache hits cost ~nothing), a settlement retry for transient facilitator hiccups, and large-result delivery via signed file export.
  • Mainnet is a config flip (X402_NETWORK=base + a mainnet facilitator) once the above are in.
A note on terms
Reselling BigQuery compute may touch Google Cloud's terms and individual dataset licenses — worth a read before pointing real money at it. (Notably, Google itself is a premier member of the x402 Foundation.)
